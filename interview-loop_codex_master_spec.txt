==================================================
0. EXECUTIVE SUMMARY
==================================================

Build a production-quality local-first web app called Interview Loop.

Interview Loop is an interview simulator and coaching app. It ingests the candidate’s real materials, generates a role-aware interview plan, runs a realistic text interview using configurable interviewer personas, stores all data locally in SQLite, and produces actionable coaching after each session.

The app is NOT a generic chatbot and NOT a platform. It is a sharp vertical product:
- create candidate profile
- upload resume
- paste job description
- optionally add cover letter/application answers/company context
- choose interview type/persona/difficulty/interest level/duration
- generate plan
- run interview
- store transcript
- score session
- generate review
- track progress

Core requirement from owner:
- Gemini integration for everything AI-related
- local database only
- web app
- speed matters
- hardcode wherever reasonable if it reduces build time without harming the demo

Run at DB startup:
- PRAGMA journal_mode = WAL;
- PRAGMA foreign_keys = ON;
- PRAGMA synchronous = NORMAL;

- PRAGMA busy_timeout = 5000;

==================================================
13. SESSION PACING RULES
==================================================

Hardcode:
- 5 min session => max 4 main questions
- 10 min session => max 6 main questions
- 15 min session => max 8 main questions

Follow-up caps:
- max 2 follow-ups per main question
- if answer is vague/incomplete/unsupported => follow-up
- if answer is sufficient => move on

Hard end conditions:
- max main question budget reached
- less than or equal to 90 seconds remaining
- user explicitly clicks end session
- user sends two consecutive empty/near-empty answers

Suggested answer length heuristic:
- short answer threshold = under 180 characters
- adequate answer threshold = over 300 characters
- but do not rely on length only; combine with model reasoning

==================================================
14. RUBRIC (HARDCODED)
==================================================

Rubric dimensions:
- clarity
- relevance
- evidence
- structure
- role_fit
- confidence

Weights:
- clarity: 0.18
- relevance: 0.22
- evidence: 0.20
- structure: 0.14
- role_fit: 0.18
- confidence: 0.08

Overall formula:
overall =
  clarity * 0.18 +
  relevance * 0.22 +
  evidence * 0.20 +
  structure * 0.14 +
  roleFit * 0.18 +
  confidence * 0.08

Score bands:
- 90-100 => excellent
- 75-89 => strong
- 60-74 => workable
- 45-59 => weak
- 0-44 => poor

Weakness trigger rules:
- evidence < 60 => needs more concrete proof
- structure < 60 => answer lacks clear narrative
- role_fit < 60 => does not map enough to target role
- relevance < 60 => drifts from actual question
- clarity < 60 => too abstract or too verbose
- confidence < 60 => delivery is under-assertive or hesitant

==================================================
15. REVIEW CATEGORIES (HARDCODED)
==================================================

Always produce:
- overall score
- dimension scores
- summary
- strengths
- weaknesses
- missed_points
- rewritten_answers
- next_steps

Rules:
- strengths: at least 2 if possible
- weaknesses: at least 2 if possible
- missed_points: 2 to 5
- rewritten_answers: maximum 3
- next_steps: 3 to 5

Selection for rewritten answers:
- choose the 3 weakest answer moments by severity
- prioritize low evidence + low role fit + low structure answers

17. PROMPT ARCHITECTURE
==================================================

Prompts must be stored in separate files under src/lib/prompts.
Do NOT inline prompt strings inside route handlers.

17.1 parseResume.ts
Goal:
- parse resume into normalized structured JSON
Rules:
- never invent employers, dates, metrics, skills, or roles
- preserve factuality
- if unclear, add to uncertainFields
- output valid JSON only

Suggested prompt content:
SYSTEM:
You are a resume parsing engine. Extract structured candidate data from the provided resume text.
Do not infer facts that are not strongly supported.
Return only valid JSON matching the schema.
If dates, company names, or metrics are ambiguous, preserve what is certain and list uncertain fields.

USER INPUT:
- raw resume text
- maybe filename
- maybe document title

17.2 analyzeJob.ts
Goal:
- extract structured role requirements
Rules:
- infer title if necessary
- identify core competencies
- identify likely interviewer concerns
- identify hidden expectations
- output valid JSON only

Suggested prompt content:
SYSTEM:
You are a hiring signal extraction engine. Analyze the job description and extract the actual skills, competencies, expectations, and likely interview themes.
Return valid JSON only.

17.3 buildPlan.ts
Goal:
- produce interview plan aligned to role, persona, difficulty, duration
Rules:
- keep it realistic
- keep it within duration
- target candidate strengths/gaps
- output valid JSON only

Hardcoded heuristics by interview type:
recruiter_screen:
- motivation
- concise background summary
- fit
- logistics/interest
- red flags

hiring_manager:
- ownership
- outcomes
- tradeoffs
- prioritization
- collaboration
- strategy/decision-making

behavioral:

17.4 liveInterviewer.ts
This is the most important prompt.

Rules:
- stay in persona
- ask one question at a time
- do not leak scoring rubric
- do not coach during the interview
- keep responses concise
- probe weak answers
- move on when answer is sufficient
- maintain consistent tone
- avoid long paragraphs
- do not output markdown bullet lists
- do not output JSON to user

Prompt should receive:
- persona config
- interest level
- difficulty
- interview type
- session objective
- current question count
- follow-up count for current topic
- last transcript turns
- retrieved context snippets
- time remaining

Output should ideally be structured internally for the app but natural for UI.
Recommended internal response schema:
{
  "agentMessage": string,
  "questionCategory": string,
  "shouldFollowUp": boolean,
  "shouldEnd": boolean,
  "reasoningTag": "new_question" | "follow_up" | "closing"
}
Even if the final user-facing text is plain, use schema internally for reliability.

17.5 evaluateSession.ts
Goal:
- grade transcript
- identify strengths/weaknesses
- cite evidence via turn indexes
Rules:
- act as evaluator, not interviewer
- score all dimensions 0-100
- tie judgments to evidence
- avoid empty generic praise
- output valid JSON only

17.6 coachSession.ts
Goal:
- turn critique into practice-ready rewrites and drills
Rules:
- generate concrete better answers
- explain why the rewrite is better
- produce next-step practice drills
- output valid JSON only

==================================================
18. DOCUMENT PARSING / CHUNKING / RETRIEVAL
==================================================

18.1 PDF parsing
Use local extraction first.
Store:
- original filename
- MIME type
- extracted raw text

If extraction fails, fail gracefully with user-facing retry guidance.

18.2 Chunking rules
Hardcode:
- target chunk size: 700 to 1000 chars
- overlap: 120 chars
- split on headings and blank lines first
- fallback to sentence boundaries
- final fallback to hard slice

Store chunk metadata:
{
  sectionLabel?: string;
  sourceFilename?: string;
  documentType: string;
  chunkIndex: number;
}

18.3 Retrieval use cases
During interview generation retrieve:
- top 2 relevant resume chunks
- top 2 job chunks
- last 4 transcript turns
- optional 1-2 prior weak-answer snippets from history if available

During scoring retrieve:
- full transcript
- job requirement snippets
- role evidence snippets from resume
- optional history context

18.4 Retrieval strategy
Do NOT use vector search.
Use:
1. keyword extraction from current question/answer
2. FTS query
3. type-based score boost
4. top-k select
5. trim to character budget

18.5 Search priorities
For live interviewer next turn:
priority order:
1. recent transcript context
2. relevant job requirement chunks
3. relevant resume chunks
4. relevant prior weaknesses if any

19. INTERVIEW ENGINE STATE MACHINE
==================================================

Implement deterministic state machine.

States:
- planned
- active_opening
- active_main
- active_followup
- closing
- completed

Transitions:
planned -> active_opening on start
active_opening -> active_main after first question
active_main -> active_followup if answer requires probing
active_followup -> active_main when resolved or cap hit
active_main -> closing when main question budget hit or time low
closing -> completed after final interviewer closing message

What to store in session-local metadata (either in session plan_json or runtime metadata_json):
- mainQuestionCount
- currentQuestionCategory
- followUpsForCurrentQuestion
- currentState
- timeRemainingEstimate
- askedQuestionIds
- latestAssessmentHints

37. INTERVIEWER RESPONSE STRATEGY
==================================================

For next turn generation:
1. save candidate turn
2. gather session config
3. gather recent transcript turns
4. retrieve relevant resume/JD snippets
5. calculate remaining question budget and follow-up budget
6. call Gemini liveInterviewer prompt
7. validate structured internal response
8. save agent turn
9. return agentMessage to client

Question selection preference:
- if current answer weak and follow-up budget remains => follow-up
- else move to next planned competency
- if nearing end => closing question or final wrap-up

4. call evaluateSession prompt
5. validate evaluation JSON
6. compute weighted overall score in app code
7. call coachSession prompt with evaluation + transcript
8. validate coaching JSON
9. persist scores
10. persist feedback items
11. mark session completed
12. return review ready

Persist feedback items by category:
- strength
- weakness
- missed_point

- next_step
